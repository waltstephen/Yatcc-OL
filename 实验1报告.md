## 一、实验过程

### 1. 实验技巧与心得

在完成词法分析器的过程中，我发现了以下技巧和心得：

- **参考 clang 的 `-cc1 -dump-tokens` 输出格式**：实验的标准答案正是通过 `clang -cc1 -dump-tokens` 生成的。通过分析其输出格式（token类型名、值、StartOfLine/LeadingSpace标记、Loc位置信息），可以精确地理解实验要求的输出规范。

- **利用预处理器指令获取源文件位置信息**：预处理后的文件中，`# linenum "filename" flags` 格式的行提供了原始源文件名和行号映射。通过在 flex 规则中解析这些指令，可以正确追踪每个 token 在原始文件中的位置。

- **Flex 的 `^` 锚点匹配行首**：使用 `^#[^\n]*` 可以精确匹配预处理器指令行，避免将代码中的 `#` 误识别为预处理指令。

- **多字符运算符优先于单字符运算符**：在 flex 规则中，`<=`、`>=`、`==`、`!=`、`&&`、`||` 等双字符运算符需要放在对应的单字符运算符 `<`、`>`、`=`、`!`、`&`、`|` 之前，利用 flex 的最长匹配规则确保正确识别。

- **关键字需要放在标识符规则之前**：由于 `{L}({L}|{D})*` 可以匹配所有关键字，必须确保所有关键字规则（如 `"int"`、`"return"` 等）在标识符规则之前定义，flex 会优先匹配前面的规则。

### 2. 最难问题与解决办法

**问题：列号追踪与 EOF 位置的正确处理**

最具挑战性的问题是正确追踪每个 token 的列号和 EOF token 的位置。

- **列号追踪**：需要在每次匹配 token 时记录其起始列号（而非结束列号），然后更新列号为 token 结束位置。这需要在 `COME` 宏中先保存当前列号作为起始位置，再增加 `yyleng` 更新列号。最初的实现中，列号在 `COME` 宏外额外增加了一次，导致列号偏移，通过重构宏定义解决了这个问题。

- **EOF 位置**：`clang` 的 `dump-tokens` 将 EOF 报告在最后一个实际 token 之后的位置（而非文件末尾的换行符之后），且不带 `[StartOfLine]` 标记。解决方案是额外记录每个非 EOF token 的结束位置（行号和列号），在遇到 EOF 时使用上一个 token 的结束位置作为 EOF 的位置。

- **预处理行的行号同步**：flex 的 `%option yylineno` 自动追踪行号，但预处理指令要求覆盖行号。关键是在 `#` 指令处理中设置 `yylineno = line - 1`，因为随后的换行符会使 flex 自动将 yylineno 加 1，从而使下一行的实际代码行号正确。

## 二、AI使用

### 1. 模型与 token 用量

- 模型：Claude Opus 4.6 (1M context)
- Token 使用量：约 80K+ tokens（包含对话和代码生成）

### 2. AI 使用心得

- **让 AI 先全面分析需求**：在编码前，先让 AI 阅读项目的全部相关文件（README、已有代码、测试脚本、评分标准），确保 AI 对完整上下文有充分理解。这比逐步提供信息效率更高。

- **利用 AI 的工具能力进行快速迭代**：AI 可以直接编译、运行测试、对比差异，形成"编码→编译→测试→修复"的快速循环，这在调试列号偏移等细节问题时非常高效。

- **关键 prompt 技巧**：最有效的 prompt 是提供明确的任务目标（"完成 task1 并提交评测"），让 AI 自主探索项目结构和需求，而非手动逐步指导。AI 能够自动发现需要修改的文件、需要添加的 token 类型、以及输出格式的精确要求。

## 三、实验建议

### 1. 实验内容建议

- 建议在 task1 的 README 中明确列出 SysY 语言需要支持的所有 token 类型（目前需要通过查看测试用例或运行 clang 来推断），这样可以减少学生在推断需求上花费的时间。
- 预处理指令的解析（获取文件名和行号）是一个较为隐蔽的需求，建议在文档中更明确地提示这一点。
